# Attestor — PDF Signature & Trust-Chain Validator

Upload a digitally signed PDF → get back an independent verdict on:

1. **Integrity** — has anything changed since it was signed?
2. **Trust** — does the signer's certificate chain to a root you trust?
3. **Revocation** — has the certificate been revoked?

This is exactly what Adobe/browser PDF viewers compute live and collapse
into a single icon. When that icon shows **"?"** instead of **"✓"**, it
almost always means step 2 failed — the signer's root CA isn't in the
viewer's default trust store. This is very common with Indian government
DSC-signed documents, because the **CCA India** root isn't bundled with
Adobe's default AATL trust list.

```
pdf-sig-validator/
├── backend/
│   ├── main.py              FastAPI app (routes)
│   ├── validator.py         Core validation logic (pyHanko)
│   ├── report_stamper.py    Appends a validation-report page to the PDF
│   ├── requirements.txt
│   └── trust_anchors/       <-- put your trusted root/intermediate certs here
└── frontend/
    └── index.html           Single-file upload UI, no build step
```

## 1. Backend setup

```bash
cd backend
python3 -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

### Add trust anchors (important)

Drop the root/intermediate CA certificates you want to trust into
`backend/trust_anchors/` as `.pem` or `.cer` files. For validating Indian
government-issued DSC signatures (e.g. Tamil Nadu CRS certificates), you
need the **CCA India root** and the issuing licensed CA's intermediate
certificate:

- Root & licensed CA certs: https://cca.gov.in/cca/?q=licensed_ca.html
- Download the root cert and the specific CA's certificate (e.g. eMudhra,
  NIC, Capricorn, Sify, depending on who issued the signer's cert), convert
  to PEM if needed, and place both in `backend/trust_anchors/`.

Without this step, every signature will validate as **intact but not
trusted** — which is the correct, honest result if you haven't configured
who you trust yet.

### Run it

```bash
uvicorn main:app --reload --port 8000
```

Health check: `GET http://localhost:8000/api/health`

## 2. Frontend setup

`frontend/index.html` is dependency-free — no npm install needed. Just:

```bash
cd frontend
python3 -m http.server 5500
```

Open `http://localhost:5500`. It calls the backend at
`http://localhost:8000` by default; override by setting
`window.ATTESTOR_API_BASE` before the script runs if you deploy the
backend elsewhere.

## 3. API

### `POST /api/validate`
`multipart/form-data`, field `file` = PDF.

```json
{
  "filename": "cert.pdf",
  "signature_count": 1,
  "overall_valid": false,
  "signatures": [
    {
      "field_name": "Signature1",
      "signer_cn": "REGISTRAR BIRTH AND DEATH",
      "issuer": "CN=eMudhra Sub CA, O=eMudhra, C=IN",
      "signing_time": "2026-07-11 10:03:12+00:00",
      "intact": true,
      "trusted": false,
      "revocation_ok": null,
      "overall_valid": false,
      "coverage": "entire_document",
      "errors": []
    }
  ]
}
```

- `intact: true, trusted: false` → this is the "?" case in Adobe: the
  document is untampered, but the signer's chain doesn't resolve to a
  configured trust anchor.
- `intact: true, trusted: true` → full "✓" case.
- `intact: false` → document was modified after signing; never trust
  regardless of chain status.

### `POST /api/validate-and-stamp`
Same input. Returns the **original PDF bytes unchanged**, plus one or
more appended pages containing a rendered ✓/✗ verdict and full chain
details — a downloadable report. It deliberately does not touch the
original signature's byte range, since altering those bytes would either
break the original signature or misleadingly forge a "valid" appearance
on a field no real signer software ever validated.

## 4. Design notes / things to harden before production

- **Revocation checking** currently uses `soft-fail` (OCSP/CRL failures
  don't block validation) so the tool still works in offline/sandboxed
  environments. For production, decide explicitly whether you want
  hard-fail behavior.
- **File size limit** is 25MB by default (`main.py`); adjust for your
  use case.
- **CORS** is wide open (`allow_origins=["*"]`) for local development —
  restrict this before deploying publicly.
- Consider adding **LTV (Long-Term Validation)** support via pyHanko if
  you need to validate signatures on certificates that have since
  expired but were valid at signing time (common for archival records).
- Consider persisting validation results with an audit log if this will
  be used for any compliance/legal workflow.

## 5. Why not just paint a ✓ into the original signature box?

That field's appearance is generated by the *original signer's* software
and is inside the cryptographically signed region of the PDF. Rewriting
it would either invalidate the real signature (integrity check fails) or,
if done outside the signed region, create a **fake, purely cosmetic**
checkmark that has no verification behind it — indistinguishable from
forgery to someone who trusts what they see on screen. This tool instead
computes a genuine, independently-reproducible verdict and presents it
alongside the untouched original.
